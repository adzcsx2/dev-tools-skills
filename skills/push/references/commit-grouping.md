# 工作区变更逻辑分组细则

> 本文件服务于 `dt:push` 的 Step 4（工作区变更分组提交）。
> 主 SKILL.md 只保留分组优先级表和执行骨架，分组算法、TDD 功能包约束、纯度校验等强制规则全部在此。
> 进入 Step 4 后必须完整执行本文件的全部强制步骤，不得只看主文件的摘要就提交。

## 前置：小批量快速路径

**变更文件总数 ≤ 3 个**：跳过下方所有分组分析，直接将全部文件合并为 **1 个 commit**，避免过度切分。文件 > 3 个时才执行完整分组流程。

## 1. 逻辑分组策略

先完整读取所有文件的 diff，再按以下规则（优先级从高到低）归并到分组：

| 优先级 | 分组规则                                                                                            | 示例场景                                           |
| ------ | --------------------------------------------------------------------------------------------------- | -------------------------------------------------- |
| P0     | **相同字符串替换**：多个文件的 diff 中出现相同的 `- old` / `+ new` 字符串对（剔除上下文行后仍重合） | 多文件统一替换 baseUrl 域名                        |
| P1     | **同一符号重命名**：函数名、类名、常量名在多文件同步变动                                            | 函数重命名后波及的所有调用方文件                   |
| P1.5   | **同一 TDD 功能包**：同一功能的实现文件、对应测试文件、需求总结文档归为同一组                       | `src/order.ts` + `order.test.ts` + `docs/order.md` |
| P2     | **同一目录 / 功能模块**：路径前缀相同且 diff 语义相关                                               | `app/api/meeting/*` 内的多个文件一同修改           |
| P3     | **同主题批量新增**：在同一次操作中新增的一组文件                                                    | 新增 skill 时一并提交 SKILL.md + README.md         |
| P4     | **独立变更**：不匹配任何上述规则的文件，回退为**单文件 commit**                                     | 彼此无关的零散修改                                 |

**分组执行逻辑**（伪代码）：

```
1. 读取所有变更文件及其 diff
2. 字符串替换检测（P0）：
   a. 提取每个文件中所有 "-旧 / +新" 行对
   b. 对 (旧, 新) 做哈希分组
   c. 出现在 ≥ 2 个文件中的 (旧, 新) 对 → 这些文件归为同一 P0 分组
3. 符号重命名检测（P1）：
   a. 对剩余文件检测标识符（函数名/类名）的统一替换
   b. 在 ≥ 2 个文件中出现同一标识符变化 → 归为 P1 分组
4. TDD 功能包检测（P1.5）：
   a. 对剩余文件识别测试文件：`*.spec.*`、`*.test.*`、`tests/**`、`test/**`、`__tests__/**`
   b. 对剩余 Markdown / 文档文件识别需求总结：`README*.md`、`docs/**`、`requirements*.md`、`spec*.md`、`design*.md`
   c. 只有当文件之间满足**显式关联**时，才归为同一 P1.5 分组；显式关联至少满足以下一项：
      - 测试文件与实现文件的归一化模块路径一致：去掉扩展名、`.test`、`.spec`、`test_`、`_test` 后指向同一模块
      - 测试 diff 中 import、调用、断言或快照名称直接引用了实现 diff 中修改过的符号 / 路由 / 命令
      - 需求总结文档 diff 至少包含 2 个与实现 / 测试 diff 共享的特征词：功能名、接口名、路由、命令、验收条件关键词
   d. 只靠"都在同一目录"或"文件名看起来相关"但没有显式关联，不能归入 P1.5
   e. 命中 P1.5 时，实现文件、对应测试、同功能需求总结必须合并成 1 个 commit
5. 目录聚类（P2）：
   a. 对剩余文件按目录前缀聚合
   b. 同目录下 ≥ 2 个文件有关联 diff → 归为 P2 分组
6. 同主题新增（P3）：
   a. 剩余新增文件中，位于同一目录或明显成套的 → 归为 P3 分组
7. 其余文件 → 各自独立 P4 单文件 commit
```

## 2. TDD 功能包约束（强制步骤）

当一次变更明显对应同一业务需求或验收场景时，必须将以下内容视为**同一个逻辑提交单元**：功能实现代码、对应测试（单元 / 集成 / E2E）、描述该功能需求或验收口径的总结文档。

**强制要求**：

- 不得因为"测试文件在 `tests/` 目录"或"文档文件在 `docs/` 目录"就把它们从功能实现中拆出去
- 不得生成"`feat: 实现功能`"后再生成"`test: 补充测试`"这种把同一 TDD 功能拆成两个 commit 的历史
- 如果测试覆盖的是同一功能需求，commit message 应优先概括该需求或用户价值，而不是单独突出"写了测试"
- 一个实现文件可以对应多个测试文件；只要这些测试文件都与该实现存在显式关联，就视为同一个 P1.5 功能包
- **纯测试**指以下两类之一：
  1. 本次变更全部是测试文件
  2. 测试文件虽然存在，但与本次任何实现文件都没有通过显式关联校验
- 只有纯测试维护、纯补充测试基建、或测试修复与任何实现改动无关时，才允许生成独立 `test` commit

## 3. 优先级与关联校验（强制步骤）

- 如果同一文件同时命中多个分组，按优先级选择：`P0 > P1 > P1.5 > P2 > P3 > P4`
- P1.5 **不使用**下方的单文件纯度门槛；改为执行显式关联校验
- 进入 P1.5 的每个文件都必须至少命中一条显式关联；否则强制移出到 P4
- 文档文件只有在与实现 / 测试共享足够特征词时才允许进入 P1.5；否则回退为独立 `docs` commit 或其他更合适的分组
- 如果一个 P1.5 组里没有"至少 1 个实现文件 + 至少 1 个显式关联的测试文件"，则该组不成立，相关文件回退到其他分组规则

## 4. 主题纯度校验（强制步骤）

**核心原则**：P0 / P1 / P2 分组不能只凭"文件里出现了主题模式"就把整个文件塞进来。一个文件只有在**绝大多数 diff 行都能被该分组主题解释**时，才允许留在分组内。否则这个文件的实际修改意图会被主题 commit 吞掉，事后从远程日志看不出它真正改了什么。P1.5 另行适用第 3 节的显式关联校验。

**对被归入 P0 / P1 / P2 的每个文件**，逐个计算纯度：

```
purity = 能被该分组主题解释的 diff 行数 / 该文件的总有效 diff 行数
```

**各分组的"主题解释行"定义**：

| 分组 | 主题解释行                                                    |
| ---- | ------------------------------------------------------------- |
| P0   | 命中该分组 `(旧串, 新串)` 替换对的所有 `-/+` 行               |
| P1   | 命中该分组符号重命名（函数名 / 类名 / 常量名）的所有 `-/+` 行 |
| P2   | 该文件 diff 中语义上能归入该目录 / 功能模块共同主题的行       |

**总有效 diff 行数的计算规则**：

- 只统计 `-` / `+` 行，不算上下文行和 `---`/`+++` 文件头
- **排除纯空白差异行**：仅缩进 / 换行 / 尾空格不同的 `-/+` 行不计入分母
- 每对 `-旧 + 新` 视为 1 个有效变更单元

**纯度门槛与处理**：

| 纯度  | 处理                                                                 |
| ----- | -------------------------------------------------------------------- |
| ≥ 30% | 文件保留在原分组                                                     |
| < 30% | **强制移出**原分组，回退到 P4 独立 commit 流程；不询问用户、不做交互 |

**边界情况**：

- 文件总有效 diff 行 ≤ 5 行：跳过纯度校验，直接留在原分组（样本太小，统计不可信）
- 文件只被 P3 / P4 分组收录：不做纯度校验
- 多个分组同时声称某文件：取纯度最高的分组；如果全部 < 30%，该文件回退到 P4

**纯度校验失败的 commit message 处理**：当文件被移出 P0 / P1 / P2 进入 P4 后，其独立 commit 的 type 和描述按该文件**残留 diff 的真实主题**生成，**不得**继续使用原分组的主题词（例如不能继续叫"统一国际化..."）。

## 5. 分组提交执行

对每个分组（含单文件 P4）依次执行：

```bash
git reset HEAD                  # 清空暂存区
git add <该分组所有文件>
git commit -m "<分组 commit message>"
```

**如果 commit 失败**（如 pre-commit hook 拒绝）→ 停止，保留当前状态，提示用户处理后手动重新执行。

## 6. Commit message 生成规则

分组 commit message 基于**该分组所有文件的聚合 diff** 生成，而非单文件 diff：

| 变更类型      | type       | 示例                                    |
| ------------- | ---------- | --------------------------------------- |
| 新增功能/文件 | `feat`     | `feat: 新增会议模块 API 接口与对应文档` |
| Bug 修复      | `fix`      | `fix: 修复登录页面输入框焦点丢失问题`   |
| 重构          | `refactor` | `refactor: 重构网络请求拦截器结构`      |
| 文档          | `docs`     | `docs: 更新 API 接口说明文档`           |
| 配置/构建     | `chore`    | `chore: 更新依赖版本`                   |
| 样式/资源     | `style`    | `style: 更新登录页面布局样式`           |
| 纯测试改动    | `test`     | `test: 补充订单模块回归用例`            |

**P0 字符串替换组**专属描述格式：`chore: 统一 <变更对象> 为 <新值>` 或 `refactor: 将 <旧值> 重命名为 <新值>`

**P1.5 TDD 功能包**专属规则：

- commit type 由**实现 / 需求本身**决定，优先使用 `feat`、`fix`、`refactor`
- 只有该组全部 diff 都是测试相关时，才允许使用 `test`
- 描述优先概括功能需求、验收结果或用户可见行为；如有必要，可在末尾补一句"并补充测试"，但不要把测试拆成第二个 commit

**Commit message 要求**：

- 使用中文描述
- 根据分组整体变更内容自动判断 type
- 描述要概括整组变更的核心意图，而非列举每个文件
- 保持简洁，一行描述清楚即可
- **禁止**在 commit message 中追加 `Co-Authored-By` 行
- **禁止用分组主导主题掩盖残留 diff**：分组内每个文件都必须通过第 4 节的纯度校验

## 7. Dry-run 模式（预览）

执行 `/dt:push --preview` 时，Step 4 仅展示分组结果与生成的 commit messages，**不执行任何写入性 git 操作**：

```
[preview] 检测到 N 个逻辑分组：

分组 1（P0 字符串替换，3 个文件）
  文件：app/api/meeting.py, docs/api.md, scripts/sync_swagger.sh
  将提交：chore: 统一测试环境域名地址为 new.example.com

分组 2（P3 同主题新增，2 个文件）
  文件：skills/push/SKILL.md, skills/push/README.md
  将提交：feat: 新增 push skill 文档

分组 3（P4 独立，1 个文件）
  文件：README.md
  将提交：docs: 更新项目简介

输入 yes 确认执行，或输入 no 退出。
```
